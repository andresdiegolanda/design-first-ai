## What Was Built

A four-endpoint Spring MVC REST API for an in-memory product catalog:
- `ProductController` — HTTP contract, delegates immediately to service
- `ProductService` — business logic, `ConcurrentHashMap` store
- `Product` / `CreateProductRequest` — validated records
- `ProductNotFoundException` — domain exception
- `GlobalExceptionHandler` — maps exceptions to HTTP 404 / 400
- `ErrorResponse` — standardised error body
- 8 service tests (plain JUnit), 9 controller tests (`@WebMvcTest`)

---

## What the Implementation Guide Prevented

Two corrections were made during impl-guide iteration — before any code was written.

| Problem | Caught at | Cost if caught at execution |
|---------|-----------|----------------------------|
| Unnecessary `ProductRepository` interface + `InMemoryProductRepository` | Components section | 2 files + Spring bean + injection wiring + refactor |
| `Long` IDs instead of `UUID` | Contracts section | Every method signature, every test, every URL example — full rewrite |

The `Long`-to-UUID catch is the more consequential one. Copilot defaulted to `Long` —
a legitimate Java convention, not a mistake in isolation. The constraint "UUID only" in
`.github/copilot-instructions.md` made it visible before a single line of code existed.

Both corrections required one iteration of the impl-guide. Neither required touching
production code.

---

## What the Layer 0 Files Added

Two Design Constraints drove both corrections:

- `.github/copilot-layer-0-architecture.md`: "Do not add a repository layer — service owns the store directly"
- `.github/copilot-instructions.md` Non-Negotiables: "UUID for all product identifiers — never Long or int"

Both corrections took one exchange each. Without the Layer 0 files, both would have
required explanation from scratch — and the contracts correction might not have surfaced
until the code was already written.
